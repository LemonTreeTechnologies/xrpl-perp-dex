# Процедура продакшен-деплоя — 3 независимых оператора

**Статус:** Черновик для обсуждения. Ещё не реализовано. Документ описывает целевую архитектуру продакшен-деплоя, когда FROST 2-of-3 пороговая подпись управляется тремя независимыми людьми (рабочее предположение: Андрей, Алекс, Том), каждый контролирует свой SGX-узел, и ни у одного оператора нет доступа к чужому серверу.

**Английская версия:** `deployment-procedure.md`. Этот документ должен оставаться синхронизированным с английской версией согласно политике bilingual docs.

---

## 1. Модель угроз

От чего защищаемся, в порядке приоритета:

1. **Один rogue-оператор.** Один из трёх операторов пытается выкатить вредоносный бинарь на свой узел и использовать его для извлечения ключей, подписи неавторизованных XRPL-транзакций или кражи средств пользователей. FROST 2-of-3 уже митигирует *подписную* сторону этой угрозы — один rogue-узел не может подписать в одиночку. Но *деплойная* сторона должна обеспечивать то же свойство: один rogue-оператор не может в одиночку поменять то, что крутится в сети.

2. **Скомпрометированная рабочая станция оператора.** Лэптоп оператора скомпрометирован (вредонос, украденные креды, evil maid). Атакующий может сделать SSH на узел этого оператора, но не должен иметь возможности залить релиз, который двое других не одобрили.

3. **Скомпрометированная среда сборки.** Билд-машина (или CI) скомпрометирована и производит бинарь, не соответствующий исходникам. Это сценарий SolarWinds.

4. **Скомпрометированный хостинг-провайдер.** Hetzner / Azure компрометируют конкретную VM (или их к этому юридически принуждают). Два других узла должны продолжать работать безопасно; аттестация скомпрометированного узла должна провалиться, и его исключат из FROST-кворума.

5. **Принуждение одного оператора.** Оператора заставляют (юридически, физически, социально) выкатить определённый бинарь. Требование 2-of-3 одобрений гарантирует, что в одиночку он не справится.

**Вне области этого документа:**
- Одновременная компрометация двух или более операторов (2-of-3 trust assumption нарушено — никакая процедура деплоя не спасёт).
- Аппаратные атаки на сам SGX (отдельная модель угроз, покрывается политикой аттестации).
- Первоначальная DKG-церемония (отдельный документ — этот документ покрывает обновления *уже работающих* узлов).

## 2. Базовые принципы

Система деплоя должна удовлетворять всем пунктам:

1. **Никакого крест-доступа к серверам.** У оператора A есть SSH/sudo только на сервере A. У B — только на B. У C — только на C. Нет общего deploy bot аккаунта, нет общего SSH-ключа, нет оператора с доступом ко всем трём узлам.

2. **2-of-3 одобрение релиза обязательно.** Ни один бинарь не деплоится *ни на один* узел, пока минимум двое из трёх операторов независимо не одобрят его, подписав хеш своим аппаратным ключом. Это должно проверяться *на каждом узле* самим узлом перед сменой бинаря — а не центральным оркестратором (который сам был бы single point of compromise).

3. **Воспроизводимые сборки.** Два оператора, собирающие один и тот же коммит на разных машинах, должны получить байт-в-байт одинаковые бинари. Без этого "я подписал хеш X" и "ты подписал хеш Y" становятся несравнимыми, и вся схема одобрения разваливается.

4. **Одобрение через аппаратные ключи.** Подписи одобрения должны исходить от аппаратного ключа, который оператор физически держит в руках (YubiKey, NitroKey или эквивалент). Подпись софтверным ключом с лэптопа нарушает принцип 2 — её можно эксфильтрировать.

5. **Аудируемая история.** Каждый релиз, каждое одобрение, каждый деплой записываются в append-only лог, который все три оператора могут независимо проверить. Никаких тихих откатов, никаких незаписанных хотфиксов.

6. **Автоматизация всего, кроме шага человеческого одобрения.** Сборка, хеширование, скачивание, проверка подписей, смена бинаря, рестарт — всё автоматизировано. Подпись релиза аппаратным ключом — *единственное* ручное действие человека, и оно остаётся ручным сознательно.

## 3. Компоненты

### 3.1 Воспроизводимые сборки

Нам нужны байт-в-байт идентичные бинари из одного исходника на разных машинах. Текущее состояние репо:

- `rust-toolchain.toml` уже пинит Rust 1.88.0 ✅
- `Cargo.lock` закоммичен ✅
- Среда сборки *не* детерминирована — зависит от системных либ, версии glibc, путей сборки. ❌

**Что нужно починить, прежде чем эта схема станет жизнеспособной:**
- Сборка внутри фиксированного контейнерного образа (конкретная версия Debian/Ubuntu, запиненная по SHA256-дайджесту, не по тегу).
- Вырезать пути сборки (`--remap-path-prefix`), вырезать timestamps, запинить `SOURCE_DATE_EPOCH`.
- Проверить воспроизводимость: два оператора собирают коммит `X` на разных машинах, хеши должны совпасть точно. Добавить как CI-гейт.
- Для SGX-энклейва: MRENCLAVE должен быть воспроизводим. SGX SDK это поддерживает в детерминированном режиме при подписи известным ключом. Требует отдельного прохода валидации.

**Критерий приёмки:** все три оператора могут независимо получить идентичные `sha256(perp-dex-orchestrator)` и идентичный `MRENCLAVE` из одного git-коммита, без согласования между собой.

### 3.2 Аппаратные ключи

Рекомендуется: **YubiKey 5 series** (PIV или OpenPGP slot), по одному на оператора, по две физические единицы на оператора (основная + backup в отдельном физическом месте).

Почему YubiKey:
- Зрелый, проаудированный, широко применяется.
- Поддерживает и PIV (X.509 + ECDSA), и OpenPGP режим.
- Требование touch-to-sign блокирует тихую подпись из-под вредоноса.
- Работает с SSH (ssh-agent), Git (подпись коммитов) и произвольной подписью файлов (gpg, age, minisign).

Каждый оператор генерирует свою подписную ключевую пару *прямо на YubiKey* — приватный ключ никогда не существует за пределами защищённого элемента. Публичные ключи коммитятся в репо в `deploy/operators/<operator>.pub`. Этот файл сам подписан всеми тремя при первичной настройке, чтобы список публичных ключей нельзя было тихо подменить.

**Backup-ключ:** у каждого оператора есть второй YubiKey с *тем же* подписным ключом (зашит в один и тот же момент, на одной air-gapped церемонии), хранящийся в отдельном физическом месте. Если основной утерян — активируется backup. Потерял оба = оператор инициирует процедуру ротации (раздел 7).

### 3.3 Манифест релиза

Манифест релиза — это маленький подписываемый файл, который говорит:

```
release_version: 1.4.2
git_commit: abcdef1234...
orchestrator_sha256: 9f8e7d...
enclave_mrenclave: 1234abcd...
enclave_mrsigner: 5678ef01...
build_timestamp: 2026-04-15T10:30:00Z
notes: "fix vAMM rebate accounting (issue #142)"
```

Именно этот манифест операторы подписывают аппаратными ключами. Подписи (по одному файлу на оператора) хранятся рядом:

```
releases/1.4.2/manifest.toml
releases/1.4.2/signatures/andrey.sig
releases/1.4.2/signatures/alex.sig
releases/1.4.2/signatures/tom.sig
```

Манифест и подписи живут в **отдельном публичном репозитории** (например, `LemonTreeTechnologies/xrpl-perp-dex-releases`). Он *отделён* от исходного репо, чтобы одобрение релиза нельзя было тихо подмешать в коммиты с кодом, и чтобы релиз-репо мог иметь более жёсткую защиту веток (no force-push, обязательная подпись коммитов).

### 3.4 Per-node deploy agent

На сервере каждого оператора крутится небольшой **deploy agent** — крошечный бинарь на Rust или Go, написанный один раз, тщательно проаудированный и почти никогда не меняющийся. Его задачи:

1. Периодически опрашивать релиз-репо на предмет новых манифестов.
2. Когда появляется новый манифест — скачать его.
3. Проверить, что есть минимум 2 валидных подписи от известных операторских публичных ключей.
4. Скачать бинарь по content-addressed URL (URL содержит sha256).
5. Проверить, что sha256 бинаря совпадает с манифестом.
6. Проверить, что MRENCLAVE SGX-энклейва совпадает с манифестом.
7. Аккуратно остановить работающий orchestrator (drain ордеров, завершить in-flight settlement).
8. Атомарно сменить бинарь.
9. Перезапустить и проверить health через локальный health-эндпоинт.
10. Записать деплой в локальный аудит-лог и отрепортить успех/провал в релиз-репо (подписав ключом аттестации узла, не операторским ключом).

У deploy agent нет операторского взаимодействия во время штатного деплоя. Единственное действие оператора — подпись манифеста часами или днями ранее.

**Сам deploy agent должен деплоиться по-другому** — его собственные обновления редки, ставятся вручную каждым оператором на свой узел, после face-to-face или видео-обсуждения. Это trusted base.

### 3.5 Кросс-аттестация

После деплоя каждый узел проверяет своих пиров через DCAP-аттестацию, прежде чем заново войти в FROST signing quorum. Если узел A поднялся с неожиданным MRENCLAVE — узлы B и C откажутся участвовать с ним в раундах подписи. Это вторая линия обороны: даже если deploy gate каким-то образом обошли на одном узле, кластер его исключит.

Это требует, чтобы пост-хакатонные работы по `feedback_dcap_subprocess_pattern.md` и `feedback_dcap_azure_two_bugs.md` стали продакшен-стабильными, и чтобы remote attestation работала на выбранном для прода облачном провайдере.

## 4. Процедура первичной настройки (one-time)

Делается один раз, желательно очно (или через защищённый видеозвонок с шерингом экрана для свидетеля).

### Шаг 1 — Церемония выдачи аппаратных ключей

Все три оператора в одной комнате (или на записываемом видеозвонке). Каждый оператор:

1. Распечатывает две новые YubiKey (основной + backup) из запечатанной упаковки, свидетели подтверждают целостность упаковки.
2. Генерирует свежую PIV/OpenPGP ключевую пару на основном YubiKey. Свидетели подтверждают, что ключ не покидает устройство.
3. Клонирует материал ключа на backup YubiKey (единственная возможность — после этого оба устройства запечатываются).
4. Экспортирует публичный ключ на USB-флешку.
5. Кладёт backup YubiKey в tamper-evident конверт, подписанный всеми тремя, увозит в отдельное физическое место после церемонии.

Три публичных ключа коммитятся в исходный репо в `deploy/operators/{andrey,alex,tom}.pub` одним коммитом, подписанным всеми тремя операторскими ключами. Этот коммит — **trust anchor**, к которому каждая последующая верификация релиза цепляется по цепочке.

### Шаг 2 — Верификация воспроизводимости сборки

Все три оператора независимо:
1. Клонируют исходный репо на выбранном коммите.
2. Собирают через канонический контейнерный образ.
3. Считают sha256 бинаря orchestrator и MRENCLAVE энклейва.
4. Сравнивают результаты друг с другом.

Если хеши не совпали — воспроизводимость сломана, надо чинить, прежде чем двигаться дальше. Это hard gate.

### Шаг 3 — Установка deploy agent

Каждый оператор на своём сервере вручную:
1. Ставит бинарь deploy agent (собранный через тот же воспроизводимый билд-пайплайн, хеш проверен).
2. Конфигурирует агент с URL релиз-репо и путём к директории операторских публичных ключей.
3. Запускает агент под systemd с ограниченным пользователем.
4. Проверяет, что он опрашивает релиз-репо и корректно простаивает, когда нового манифеста нет.

### Шаг 4 — End-to-end dry run

Прогнать no-op релиз (например, версию `0.9.9-test`) через весь пайплайн:
1. Сборка, хеш, манифест.
2. Каждый оператор подписывает манифест YubiKey'ем, коммитит подпись в релиз-репо.
3. После 2-of-3 подписей — смотреть, как каждый deploy agent независимо скачивает, проверяет и (в dry-run режиме) репортит, что *он бы задеплоил*.
4. Сделать настоящую смену бинаря на staging-инстансе (не на проде), чтобы убедиться в graceful drain и рестарте.

Только после успешного dry run система считается готовой к настоящим релизам.

## 5. Процедура каждого релиза (steady state)

Эту процедуру вы прогоняете каждый раз, когда нужно выкатить новую версию в прод.

### Стадия A — Сборка и предложение

Делает **тот оператор, который вносит изменение** (может быть любой из трёх; не привилегированная роль).

1. Пушит изменение как обычный PR в исходный репо. Проходит ревью и мерджится в `main` как обычно.
2. Тегает релиз: `git tag -s v1.4.2` (подписанный git tag, аппаратным ключом оператора).
3. Запускает канонический билд-пайплайн. На выходе: бинарь orchestrator, .so энклейва, MRENCLAVE.
4. Генерирует манифест релиза (`manifest.toml`) с хешами.
5. Загружает бинарь + энклейв в content-addressed хранилище (URL вида `https://releases.example/{sha256}`, так что сам URL верифицируем).
6. Открывает PR в релиз-репо, добавляющий `releases/1.4.2/manifest.toml`. Этот PR НЕ должен содержать ни одной подписи — они приземляются отдельно, см. стадию B.

### Стадия B — Независимая верификация и подпись

Каждый из двух других операторов (и в идеале сам предлагающий тоже, для полного 3-of-3 в нормальных случаях):

1. Пуллит PR с манифестом.
2. Чекаутит исходник на коммит, упомянутый в манифесте.
3. **Независимо собирает** бинарь на своей машине. Воспроизводимая сборка означает, что его хеш должен точно совпасть с хешем в манифесте.
4. Проверяет, что MRENCLAVE энклейва совпадает.
5. Читает фактический diff vs предыдущий релиз. Задаёт вопросы, если что-то непонятно.
6. Если устраивает — подписывает манифест:
   - YubiKey вставлен.
   - `signify -S -s yubikey:slot -m manifest.toml -x signatures/<operator>.sig` (или эквивалент для выбранного инструмента).
   - YubiKey мигает, оператор касается, чтобы подтвердить.
7. Коммитит файл с подписью в PR релиз-репо.

Когда есть 2-of-3 подписей — PR с манифестом мерджится.

**Критично:** шаг верификации (3) не подлежит обсуждению. Оператор, подписывающий без независимой сборки, — это вектор атаки single-operator-trust. Вся схема держится на том, что минимум двое операторов независимо подтвердили соответствие бинаря исходникам.

### Стадия C — Автоматизированный rollout

После того как манифест замерджился в `main` ветку релиз-репо, никаких ручных действий не нужно. Каждый deploy agent на каждом сервере:

1. Обнаружит новый манифест в пределах своего интервала опроса (например, 60с).
2. Локально проверит подписи (offline, против trust-anchor публичных ключей, зашитых при настройке).
3. Скачает бинарь и энклейв из content-addressed хранилища.
4. Проверит хеши.
5. Drain → swap → restart → health check.
6. Проведёт кросс-аттестацию с другими узлами. Когда все три узла отрепортят здоровую аттестацию, signing quorum возобновится.

**Rolling vs synchronised:** предпочтение **rolling** (по одному узлу за раз, с 5-минутным soak между ними), чтобы если релиз ломается — это обнаружилось на узле A, а B и C ещё на старой версии и кластер продолжает подписывать. Реализация: поле `rollout_order` в манифесте, кодирующее, какой узел деплоится первым / вторым / третьим, и агенты ждут своей очереди, наблюдая за квитанцией о деплое предыдущего узла в релиз-репо.

### Стадия D — Пост-деплойная проверка

Каждый оператор независимо проверяет здоровье своего узла (логи, метрики, статус аттестации). Если какой-то узел отрепортил провал — включается процедура **отката** (раздел 6).

## 6. Откат и реакция на инцидент

### Автоматический откат

Deploy agent держит предыдущий бинарь на диске. Если после смены:
- Health check не проходит N секунд подряд, ИЛИ
- Кросс-аттестация с пирами не проходит, ИЛИ
- Orchestrator паникует на старте,

агент автоматически возвращается к предыдущему бинарю, перезапускается и постит "rollback" квитанцию в релиз-репо (подписанную ключом аттестации узла). Два других узла видят это и понимают, что один пир на старой версии.

### Ручной откат

Если релиз задеплоился успешно, но в проде проявляется баг (например, неправильный учёт fee, видный только после реальных торгов) — операторы могут задеплоить более старый релиз:

1. Открыть новый PR с манифестом, указывающим на хеши более ранней версии (пересборка не нужна, хеши уже известны).
2. 2-of-3 подписать как обычно.
3. Rollout идёт штатно.

Откат рассматривается как обычный релиз. Нет специального "emergency rollback" пути — весь смысл 2-of-3 в том, что ни один оператор не имеет emergency override.

### Утерянный аппаратный ключ

Оператор потерял основной YubiKey. Активирует backup (хранящийся в отдельном физическом месте). Изменения в протоколе не нужны.

Если оператор потерял *оба* — основной и backup — он не может подписывать релизы. Двое других могут продолжать пушить релизы (2-of-3 всё ещё удовлетворено), а процедура ротации (раздел 7) заменяет недостающий ключ.

### Принуждение оператора

Если оператора принуждают подписать вредоносный релиз — он должен отказаться. Двух других одновременно принудить нельзя (вне модели угроз). Один принуждаемый оператор не может выкатить релиз. Если он раскрывает факт принуждения двум другим — те просто не подпишут, и вредоносный релиз никуда не уйдёт.

Кнопка "паника" / duress code на этом уровне *не* рекомендуется — это разумно на уровне локального разблокирования ключа оператором (PIV PIN с отдельным "wipe PIN"), но добавляет сложность на уровне манифеста с малой пользой, поскольку требование 2-of-3 уже защищает протокол.

## 7. Ротация операторов

Добавление или удаление оператора (например, Том передаёт дела новому человеку, или один из троих отходит):

1. Замещающий оператор проходит церемонию выдачи аппаратных ключей (раздел 4 шаг 1) в присутствии двух оставшихся операторов.
2. Двое оставшихся подписывают запись в `deploy/operators/CHANGES`, добавляющую новый публичный ключ и удаляющую старый.
3. Это изменение само по себе является манифестом релиза, деплоится через обычный пайплайн (2-of-3 подпись с *текущим* набором операторов, включая уходящего, если он доступен).
4. После деплоя deploy agents на всех трёх узлах обновляют свой trust-anchor набор публичных ключей.
5. С этого момента подпись нового оператора признаётся, а старого — нет.

Если уходящий оператор не сотрудничает или скомпрометирован, двое оставшихся всё равно могут провести ротацию (2-of-3 удовлетворено и без него), при условии что сама ротационный манифест проходит как обычный релиз.

## 8. Что можно и что нельзя автоматизировать

| Шаг | Автоматизируемо? | Почему |
|---|---|---|
| Сборка из исходников | Да | CI запускает воспроизводимый билд-пайплайн |
| Генерация манифеста | Да | Подсчёт хеша детерминирован |
| Сбор подписей | Нет | Аппаратный ключ + касание человека — это trust gate |
| Мердж манифеста в релиз-репо | Частично | Auto-merge, как только есть 2-of-3 подписей |
| Получение нового манифеста deploy agent'ом | Да | Polling |
| Проверка подписей на узле | Да | Криптография, ввода человека не требуется |
| Проверка хеша бинаря | Да | Криптография |
| Drain / swap / restart | Да | Стандартное управление процессами |
| Порядок rollout | Да | Закодирован в манифесте |
| Кросс-аттестация | Да | Механика DCAP |
| Откат при провале health check | Да | Локальное решение агента |
| Откат при проде-баге | Нет | Требует человеческого решения + 2-of-3 подписи |

Единственный шаг, остающийся ручным, — подпись человеком через аппаратный ключ. В этом и смысл — это гейт, который мешает доверять остальной автоматизации в одностороннем порядке.

## 9. Открытые решения, которые нужно принять до реализации

Это вещи, по которым у нас пока нет твёрдого мнения и которые надо обсудить:

1. **Инструмент подписи.** `signify` (OpenBSD, очень простой) vs `minisign` (signify-совместимый, шире пакетирован) vs `gpg` (полнофункциональный, но сложный) vs `cosign` (sigstore-экосистема, интегрируется с OCI-реестрами). Рекомендация: **minisign** ради простоты, с YubiKey, держащим приватный ключ через bridge OpenPGP-слота, ИЛИ **cosign**, если мы в итоге всё равно поставляем контейнерные образы.

2. **Хостинг релиз-репо.** GitHub (удобно, но SPOF), self-hosted Gitea (больше контроля), или не-git append-only лог (sigstore Rekor, или просто signed-append-only-file на сервере каждого оператора с гossip). Рекомендация: **GitHub** с защитой веток на старте, плюс зеркало на self-hosted инстанс для устойчивости.

3. **Content-addressed хранилище бинарей.** S3-совместимое ведро с public read (дёшево, просто, но хостер видит, что вы раздаёте) vs IPFS (децентрализовано, но операционно тяжелее) vs маленький файл-сервер на каждом операторском узле с кросс-репликацией. Рекомендация: **S3-совместимое ведро** (например, Cloudflare R2) для простоты — content-addressing означает, что хранилищу не нужно доверять.

4. **Контейнер vs bare-metal деплой.** Сейчас Hetzner запускает orchestrator под nohup. Прод должен как минимум крутиться под systemd, в идеале внутри контейнера с тугим seccomp/apparmor профилем. Процедура deploy agent предполагает смену бинаря; смена контейнера тоже подходит, выберите одно и держитесь.

5. **Частота валидации воспроизводимости.** Как часто мы прогоняем "все три оператора пересобирают последний релиз и подтверждают совпадение хешей" как аудит? Каждый релиз? Раз в месяц? Рекомендация: **каждый релиз**, поскольку это буквально предусловие схемы.

6. **Ревью кода deploy agent.** Поскольку это самый security-критичный кусок (он обеспечивает 2-of-3 гейт на каждом узле), он должен быть маленьким, проаудированным и сам собираться воспроизводимо. Кто пишет, кто ревьюит, когда?

7. **Начальный seed доверия.** Операторские публичные ключи, закоммиченные при настройке, доверены, потому что все три оператора были физически в одном месте. Как мы доказываем это *самим себе* через год? Рекомендация: подписанная видеозапись церемонии, хеши публично опубликованы.

## 10. Предполагаемый порядок реализации

Исходя из "у нас много времени" — не в режиме хакатонной гонки:

1. **Воспроизводимые сборки** в первую очередь. Без этого ничего не работает. Это в основном инженерия, без криптографии.
2. **Тулинг для аппаратных ключей** — выбрать инструмент подписи, написать обёрточные скрипты, прогнать бумажное упражнение по флоу подписи.
3. **Релиз-репо + формат манифеста** — определить формат файла, trust-anchor коммит, логику верификации.
4. **Deploy agent v1** — маленький, сфокусированный, тщательно проаудированный.
5. **Single-node dry run** — доказать, что агент работает на одной машине, прежде чем подключать остальных.
6. **Three-operator dry run** — полная церемония, полный пайплайн, без продакшен-трафика.
7. **Интеграция кросс-аттестации** — подключить существующие наработки по DCAP в стартап-проверки deploy agent.
8. **План продакшен-выкатки** — мигрировать с текущего nohup-деплоя на новый пайплайн.
9. **Drill ротации** — отрепетировать удаление и повторное добавление оператора, прежде чем это понадобится в реальности.

На каждом шаге предыдущие шаги работают в фоне и проходят боевое тестирование.

## 11. Runbook для hotfix на mainnet

Эта секция — конкретная пошаговая процедура, которую выполняет дежурный оператор, когда новый релиз уже подписан 2-of-3 (секция 5) и должен быть выкачен на кластер, уже удерживающий реальные пользовательские средства на XRPL mainnet. Всё, что выше в документе — о том, *как мы решаем что деплоить*; эта секция — о том, *как мы это деплоим, не теряя state и не теряя средства*.

### 11.1 Инвариант: MRSIGNER sealing недоступен

Каждый sealed blob на каждой продакшен-ноде (FROST key share, таблица tx dedup, recovery-артефакты) запечатан на **MRENCLAVE** того энклейва, который его записал. Это сделано намеренно: MRSIGNER sealing был оценён и отклонён — см. `deployment-dilemma.ru.md` §"Стратегия 2: MRSIGNER-based Sealing — ОТКЛОНЕНА" для полной аргументации (верификаторы аттестации проверяют MRENCLAVE, не MRSIGNER; one-way door; компрометация build pipeline → полная экстракция ключей).

Операционное следствие этого инварианта:

> **Любое изменение, порождающее новый MRENCLAVE, не может распечатать старый state.** Для энклейва нет пути апгрейда через file swap. Каждое такое изменение — on-chain ротация ключей через `SignerListSet`, а не замена бинаря.

Это самая важная вещь, которую нужно усвоить перед запуском runbook'а. Если ловите себя на мысли "давайте просто скопируем новый `.signed.so` на место" — остановитесь и перечитайте эту секцию.

### 11.2 Дерево решений: по какому пути я иду?

До того как трогать что-либо в продакшене, классифицируйте hotfix:

```
Релиз изменил что-либо из:
  - Enclave/*.cpp, Enclave/*.h, Enclave/*.edl
  - PerpState.h или любой struct, персистимый внутри энклейва
  - Enclave/Enclave.config.xml (TCS, heap, stack, debug flag)
  - Версию SGX SDK или build flags, влияющие на энклейв
  - Ключ подписи энклейва
?

Нет → Путь A: hotfix только orchestrator'а
Да  → Путь B: изменение энклейва (новый MRENCLAVE)
```

Быстрая санити-проверка: `sha256sum enclave.signed.so` на релизном артефакте vs на текущей запущенной ноде. Если хеши совпадают — это определённо Путь A. Если разные — Путь B, точка; исключений "ну это крошечное изменение энклейва" не существует.

### 11.3 Pre-flight (оба пути)

Пройдите этот список до того, как тронете любую продакшен-ноду. Если хотя бы один пункт не выполнен — прервать и починить сначала; варианта "разберёмся во время окна" нет.

1. **У релизного манифеста 2-of-3 валидные подписи** (секция 5). Проверить каждую подпись локально против trust-anchor pubkeys.
2. **Хеш воспроизводимой сборки совпадает** — вы независимо собрали от тегированного коммита и ваш `sha256` совпадает с манифестом.
3. **Определён предыдущий хороший манифест** — запишите его версию и commit hash. Это цель отката; не ищите её под давлением.
4. **Пройден testnet soak** — ровно эта сборка крутится на testnet-кластере ≥24 часов со здоровыми FROST раундами.
5. **Снят снимок состояния кластера:**
   - Текущий MRENCLAVE на каждой ноде (с локального эндпоинта аттестации).
   - Текущий XRPL account sequence escrow-аккаунта.
   - Текущий on-chain signer list (адреса + quorum).
   - Баланс XRP на escrow и все открытые позиции.
   - FROST health: timestamp последнего успешного quorum-раунда.
6. **Нет in-flight выводов** — проверить таблицу pending-tx в orchestrator'е. Если непусто — либо дождаться подтверждения, либо прервать.
7. **Нет неподтверждённых депозитов моложе 20 ledgers** — всё в mid-validation должно устояться до старта.
8. **Все трое операторов доступны** в общем канале на время окна. Ротация по пути B требует живых 2-of-3 FROST раундов по ходу процедуры.
9. **Релиз отката уже собран и хеш-верифицирован** на диске каждой ноды как `.prev`. Откат не должен зависеть от пересборки.

### 11.4 Путь A — hotfix только orchestrator'а

Энклейв крутится всё время. Меняется только бинарь orchestrator'а. Sealed data не трогаем. Ключи не ротируем. On-chain действий нет.

Порядок нод: **Hetzner первым** (канарейка — ощущается "наиболее своей"), потом Azure node-1, node-2, node-3, с 5-минутным soak между каждой.

Шаги на ноду:

1. Подтвердить `sha256sum enclave.signed.so` на этой ноде совпадает с продакшен-манифестом. Если нет — это не Путь A, стоп.
2. Остановить orchestrator:
   ```
   sudo systemctl stop perp-dex-orchestrator-prod
   ```
3. Проверить, что host-процесс энклейва на порту 9088 всё ещё живой (`pgrep -f enclave_signed`). Если энклейв умер вместе с orchestrator'ом — что-то не так, прервать до продолжения.
4. Переместить новый бинарь, сохранив предыдущий как `.prev`:
   ```
   sudo mv /opt/perp-dex/bin/orchestrator /opt/perp-dex/bin/orchestrator.prev
   sudo install -m 755 /tmp/orchestrator-new /opt/perp-dex/bin/orchestrator
   ```
5. Запустить orchestrator:
   ```
   sudo systemctl start perp-dex-orchestrator-prod
   ```
6. Health check должен возвращать OK непрерывно 60s:
   ```
   for i in $(seq 1 60); do curl -fsS http://localhost:3003/v1/health || exit 1; sleep 1; done
   ```
7. Прогнать **dry-run FROST round** против двух других peers (без сабмита в XRPL — в orchestrator'е есть диагностический эндпоинт). Раунд должен завершиться в пределах 10s.
8. Soak 5 минут. Смотреть `journalctl -u perp-dex-orchestrator-prod -f` на ошибки.
9. Если чисто — переходим к следующей ноде. Если что-то дёргается — см. §11.6 откат.

Общее wall-clock: ~25 минут на четыре ноды (5 мин на ноду + 5 мин inter-node soak).

### 11.5 Путь B — изменение энклейва (новый MRENCLAVE)

Это полная pre-planned key-rotation церемония. Шаги не пропускать. Прочтите `deployment-dilemma.ru.md` §"Стратегия 4: Pre-planned Key Rotation (recommended)" целиком до старта; анализ поверхности атаки (векторы 1–6) — это обоснование, зачем каждый шаг существует.

Порядок нод: **Hetzner первым** (канарейка), потом Azure node-1, node-2, node-3. Никогда более одной ноды в состоянии ротации одновременно. В кластере всегда должно быть ≥2 нод на согласованном MRENCLAVE, чтобы подписание продолжалось.

Шаги на ноду:

1. **Запустить новый энклейв на staging-порту** (9089), оставляя старый энклейв на 9088 работающим и обслуживающим живой трафик.
   ```
   sudo systemctl start perp-dex-enclave-prod-new
   ```
   Новый энклейв поднимается с пустым sealed state (пока нет FROST share, нет истории tx dedup).

2. **Новый энклейв внутренне генерирует свежий keypair** и выпускает DCAP attestation quote, связывающий `new_MRENCLAVE` с новым XRPL-адресом. Ecall: `ecall_generate_key_with_attestation`.

3. **Peers верифицируют attestation quote.** Orchestrator на ротируемой ноде посылает quote в orchestrator'ы двух других нод, которые передают его в свои (старые) энклейвы. Каждый peer-энклейв проверяет:
   - Quote цепочкой верифицируется к root Intel'а.
   - MRENCLAVE в quote совпадает с полем `enclave_mrenclave` в актуально мерженом релизном манифесте.
   - Публичный ключ в report data quote совпадает с тем, что предложил orchestrator.
   Если какой-либо peer отклоняет — прервать: новый энклейв врёт про свою идентичность. Остановить новый энклейв, **не удалять** старые данные, эскалировать.

4. **Co-sign SignerListSet #1 — добавить нового signer'а, поднять quorum до 3.**
   - Транзакция добавляет новый адрес в signer list escrow-аккаунта.
   - Quorum одновременно поднимается с 2 до **3** на время окна ротации (защита от вектора 1 из deployment-dilemma §Attack surface analysis).
   - `LastLedgerSequence` ставится в `current_ledger + 20` (~60s окно валидации).
   - 2-of-3 существующих (старых) энклейвов co-sign'ят.
   - Если не подтвердилось внутри `LastLedgerSequence`: ключ не добавлен, состояние не изменилось, прервать ротацию этой ноды и эскалировать.

5. **Co-sign SignerListSet #2 — удалить старого signer'а, опустить quorum до 2.**
   - Транзакция удаляет старый адрес, оставляя 3 signer'а (два неротированных peer'а + новый ключ этой ноды).
   - Quorum возвращается с 3 в 2.
   - `Sequence = (Sequence tx #1) + 1` — per-account sequence XRPL форсит порядок (защита вектора 4).
   - Ретрай с экспоненциальным backoff до 10 минут суммарно.
   - **Если tx #2 не подтвердилась за 10 минут:** сабмит аварийного SignerListSet #3, удаляющего *новый* ключ (возврат к исходному 3-signer списку). Лучше прервать апгрейд, чем бесконечно держать окно с 4 ключами (защита вектора 3).

6. **Проверить on-chain состояние.** Стянуть signer list из XRPL и подтвердить: ровно 3 записи, quorum 2, новый ключ присутствует, старый отсутствует.

7. **Остановить старый энклейв.**
   ```
   sudo systemctl stop perp-dex-enclave-prod-old
   ```
   В этот момент новый энклейв на 9089 — единственный энклейв на ноде, но ещё не промотирован в продакшен-порт/имя сервиса.

8. **Soak до shred (золотое правило).** Прогнать один реальный FROST раунд на mainnet (no-op: например, тривиальный escrow memo-платёж на 1 drop обратно на сам escrow, или ping-style liveness-транзакция, согласованная заранее). Дождаться подтверждения tx on-chain. Подтвердить, что новый энклейв подписал корректно. Это последнее безопасное окно — не пропускать.

9. **Шреднуть старые sealed data.**
   ```
   sudo shred -u /var/lib/perp-dex/prod/frost_share.sealed
   sudo shred -u /var/lib/perp-dex/prod/tx_dedup.sealed
   sudo shred -u /var/lib/perp-dex/prod/nonce_ctr.sealed
   sudo rm -f /opt/perp-dex/bin/enclave.signed.so.old
   ```
   (Запечатано под старый MRENCLAVE; невосстановимо даже в теории, но всё равно шредим, чтобы убрать соблазнительную фикцию "можем откатиться, восстановив старые файлы".)

10. **Промотировать новый энклейв.** Указать продакшен systemd unit на новый бинарь и порт; рестартнуть под продакшен-именем сервиса.
    ```
    sudo systemctl stop perp-dex-enclave-prod-new
    sudo mv /opt/perp-dex/bin/enclave.signed.so.new /opt/perp-dex/bin/enclave.signed.so
    sudo systemctl start perp-dex-enclave-prod
    ```

11. **Кросс-аттестация с peers.** Каждая из двух других нод прогоняет свою проверку аттестации против этой ноды и должна принять новый MRENCLAVE. Если какой-либо peer отклоняет — эта нода вне quorum'а, см. §11.7.

12. **Операционный soak — минимум 10 минут.** Смотреть:
    - Здоровые FROST раунды с участием этой ноды.
    - Нет неожиданных ошибок в `journalctl -u perp-dex-enclave-prod -u perp-dex-orchestrator-prod`.
    - Статус аттестации остаётся "OK" на всех трёх нодах.

13. **Переход к следующей ноде** только после чистого прохождения soak. Ротируем Hetzner → Azure-1 → Azure-2 → Azure-3 последовательно.

Конечное состояние после ротации всех четырёх нод: signer list из 3 новых адресов (все связаны с новым MRENCLAVE), quorum 2, тот же баланс escrow, ни одного XRP не передвинуто за церемонию.

Общее wall-clock: ~60–75 минут на весь кластер, если без откатов.

### 11.6 Критерии отката

Перечислено от наименее до наиболее драматичного. Выбирайте самый узкий, подходящий под режим отказа.

| Триггер | Scope | Действие |
|---|---|---|
| Health check Пути A падает во время soak | Одна нода | Агент автоматически откатывает на `.prev` бинарь, рестартует, постит rollback receipt. Кластер не затронут. |
| Путь B — add-signer tx не подтвердилась в пределах `LastLedgerSequence` | Одна нода | Прервать церемонию на этой ноде. Ключ не добавлен. Старый энклейв продолжает работать. Эскалация. |
| Путь B — remove-signer tx не подтвердилась за 10 минут | Одна нода | Сабмитнуть аварийный SignerListSet #3, удаляющий *новый* ключ. Возврат к исходному 3-signer списку. Эскалация. |
| Путь B — новый энклейв не проходит DCAP аттестацию у peers | Одна нода | Остановить новый энклейв. Старый продолжает работать, старые sealed data целы. Расследовать расхождение MRENCLAVE (дрейф build env, неправильный манифест) до повторной попытки. |
| Путь B — кросс-аттестация после промоушена падает у одного peer | Одна нода | Если soak ещё не шреднул старые данные: рестартовать старый энклейв из его sealed blobs. Если шред уже сделан: эта нода вне quorum, следовать §11.7. |
| Операционный сбой во время post-rotation soak (плохие FROST раунды, поломанный state) | Кластер | Co-sign SignerListSet, повторно добавляющий старый ключ предыдущей ноды и удаляющий новый ключ. Кластер возвращается к pre-hotfix signer set. Карантин плохой ноды. |
| Продакшен-баг обнаружен после полной ротации кластера | Кластер | Трактовать как новый релиз: подписать 2-of-3 манифест, указывающий на предыдущую хорошую версию. Прогнать этот runbook снова в обратную сторону. |

Ни один путь отката не предполагает, что один оператор обходит 2-of-3. Если ситуация кажется требующей этого — вы либо в out-of-scope случае "двое операторов скомпрометированы", либо неправильно классифицировали отказ.

### 11.7 Худший случай: sealed data потеряны на одной ноде

Если шаг 9 §11.5 (shred) выполнен и новый энклейв затем отказывает на этой ноде перманентно (аппаратный сбой, непойманный ранее MRENCLAVE mismatch, невосстановимая коррупция данных) — нода **не имеет FROST share**: ни старой, ни новой.

Эта нода вне FROST quorum перманентно, пока не пройдёт свежая DKG церемония. Кластер продолжает подписывать с 2-of-2 из оставшихся нод (список из 3 signer'ов, quorum 2 — математически цел), но запас прочности теперь ноль. Сбой второй ноды в этом окне заморозит escrow.

Процедура восстановления (отдельная от этого runbook):

1. Переподготовить затронутую ноду с нуля (новая VM, wipe + переустановка, или новое железо).
2. Прогнать свежую 3-party DKG с двумя здоровыми нодами + замена. Это даёт новый набор FROST shares, связанных с текущим MRENCLAVE.
3. Сабмитнуть SignerListSet, заменяющий адрес мёртвой ноды на адрес замены. Quorum остаётся 2.
4. Возобновить нормальную работу.

Сама DKG — отдельная церемония, вне рамок этого документа. Она документирована в `deployment-procedure.md` как будущее дополнение (см. секцию 1 "вне рамок").

### 11.8 Золотое правило

> **Не шредить старые sealed data, пока новый энклейв не подписал хотя бы один реальный FROST раунд на mainnet.**

Всё до shred обратимо. Shred — точка невозврата для этой ноды. 10-минутный soak в §11.5 шаг 8 существует именно чтобы у вас было последнее безопасное окно для прерывания.

Если по какой-либо причине вы не уверены, что новый энклейв действительно выдаёт валидные подписи (шум в логах, нестабильность аттестации, сетевые странности во время soak) — **не шредить**. Продлить soak, расследовать, и продолжать только когда каждый сигнал чист. Лишний час soak всегда дешевле свежей DKG.

### 11.9 Post-hotfix чеклист

После того как кластер полностью на новом релизе:

1. Все три ноды репортят новый MRENCLAVE в аттестации.
2. XRPL signer list — ровно 3 записи, quorum 2, все три — новые адреса.
3. Баланс escrow без изменений (за вычетом liveness-ping tx из §11.5 шаг 8).
4. Одна успешная реальная user-initiated операция (депозит, вывод, или изменение позиции), подписанная новым кластером.
5. Audit log запись закоммичена в релиз-репо, подписана дежурным оператором, с указанием: версия релиза, start/end timestamp, таймингов по каждой ноде, любых отклонений от этого runbook.
6. Предыдущий релиз сохранён на диске как `.prev` на стандартное rollback-окно (предлагается 30 дней), затем подчищается.

## Приложение A — Что эта схема НЕ решает

Честно про границы:

- **Сговор двух операторов.** Если Алекс и Том сговорятся — у них 2-of-3 и они могут выкатить что угодно. Это by design — trust assumption протокола в том, что меньше двух операторов вредоносны.
- **Supply chain сборки.** Вредоносная зависимость в `Cargo.lock` будет в воспроизводимой сборке у всех. Независимая верификация хешей лишь докажет, что все трое получили *одну и ту же* отраву. Митигация: ревью зависимостей при мердже, `cargo deny`, vendored deps для критических путей.
- **Supply chain YubiKey.** Бэкдоренный YubiKey может слить приватные ключи. Митигация: покупать у авторизованных реселлеров, проверять упаковку, в идеале покупать у разных источников для трёх операторов.
- **Доверие DCAP/SGX.** Весь слой аттестации доверяет Intel. Вне рамок этого документа, но стоит помнить при рассуждениях о end-to-end доверии.
