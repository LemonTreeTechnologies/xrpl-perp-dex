## Процедура обновления mainnet на Hetzner

Чеклист безопасного обновления:

1. **Сборка** на Hetzner из git (staging-ветка протестирована)
2. **Снимок** текущего состояния:
   - `cp -r /tmp/perp-9088/ /tmp/perp-9088.backup-$(date +%s)/`
   - `pg_dump perp_dex > /tmp/perp_dex_backup.sql`
3. **Остановка** mainnet: `kill <PID>` (nginx отдаёт 502)
4. **Замена** бинарника: `cp target/release/perp-dex-orchestrator /tmp/perp-9088/`
5. **Запуск** с теми же аргументами
6. **Health check**: `curl http://localhost:3000/v1/health`
7. **Проверка** XRPL mainnet: баланс эскроу не изменился
8. **Откат** если health check не прошёл:
   - `kill <PID>`
   - Восстановить бинарник из backup
   - Перезапуск

**Даунтайм:** ~30 секунд. Приемлемо при текущем трафике (около нуля).

## Сложный вопрос: обновление Enclave

Всё выше обсуждает деплой **оркестратора** (Rust). Enclave (C++ SGX) —
фундаментально сложнее из-за **привязки MRENCLAVE**.

### Почему enclave ≠ оркестратор

| Свойство | Оркестратор (Rust) | Enclave (C++ SGX) |
|----------|--------------------|--------------------|
| Состояние | PostgreSQL (внешнее) | Sealed к CPU + MRENCLAVE |
| Замена бинарника | Stop → replace → start | Новый MRENCLAVE → **все sealed ключи потеряны** |
| Откат | Скопировать старый бинарник | Sealed данные старого MRENCLAVE работают |
| Ключи | Живут в enclave, оркестратор не имеет | Ключи **и есть** состояние enclave |
| Плохой деплой | 502 на 30 сек | **Безвозвратная потеря ключей** |

### Проблема MRENCLAVE

SGX `sgx_seal_data()` привязывает данные к:
1. **MRENCLAVE** — хеш бинарника (меняется при ЛЮБОМ изменении кода)
2. **Идентификатор CPU** — per-platform ключ

Это означает:
- Новый `enclave.signed.so` (даже исправление опечатки) = **другой MRENCLAVE**
- Новый enclave **не может расшифровать** данные старого
- Приватные ключи **безвозвратно потеряны** для нового enclave
- Это by design в SGX, не баг

### Что теряется при наивном обновлении

1. Все приватные ключи enclave → XRPL-адреса из SignerListSet
2. Мультисig эскроу перестаёт работать
3. Средства пользователей заблокированы
4. FROST shares (если используются) — не переносимы между машинами

### Когда ПРИДЁТСЯ обновлять enclave

Известные триггеры (не гипотетические):

1. **FROST cross-machine share transport** — сломан (Bug 1 в
   SHARED-ENCLAVE-BUGS.md). Фикс = C++ → новый MRENCLAVE
2. **BTC signing path** — если PM понадобится
3. **`MAX_FROST_GROUPS = 4`** — слишком мало для продакшена
4. **DKG "share already received" без reset** — блокирует retry
5. **Патчи безопасности** — любой CVE в коде enclave
6. **Новые фичи** — spending limits, margin engine внутри enclave

### Стратегии обновления

#### Стратегия 1: Dual-Server миграция

Старый enclave (порт 9088) и новый (порт 9089) работают параллельно.

**Процедура:**
1. Деплой нового enclave на отдельный порт
2. Генерация нового ключа → новый XRPL-адрес
3. Старый enclave подписывает SignerListSet: добавить нового signer'а
4. Старый enclave подписывает SignerListSet: удалить старого
5. Старый enclave деактивируется

**Критично:** старый enclave должен оставаться рабочим на протяжении
всей миграции.

#### ~~Стратегия 2: MRSIGNER-based Sealing~~ — ОТКЛОНЕНА

**Отклонена.** Переключение `sgx_seal_data()` с MRENCLAVE на MRSIGNER
означает, что любой enclave, подписанный тем же ключом вендора, может
расшифровать все секреты. Это:
- Не пройдёт аудит безопасности — верификаторы аттестации проверяют MRENCLAVE, не MRSIGNER
- Создаёт дорогу в один конец: после перехода на MRSIGNER возврат к MRENCLAVE
  требует ту же ротацию ключей что и Стратегия 4, так что короткого пути нет
- Компрометация сборочного пайплайна → полное извлечение ключей

**Даже для testnet не используем.** Привычка к MRSIGNER sealing означает
расхождение testnet-кода с моделью безопасности mainnet, и это расхождение
ударит в самый неподходящий момент.

#### ~~Стратегия 3: Экспорт через recovery~~ — ОТКЛОНЕНА

Использовать `account.recovery` файлы для экспорта/импорта ключей.

**Проблема:** сам механизм recovery позволяет оператору извлечь приватный
ключ из enclave в любой момент — это фундаментальное нарушение trust model,
а не временный компромисс. Если ключ можно достать, то SGX enclave
перестаёт быть trust boundary.

**Вердикт:** отклонена. Нарушает модель доверия по определению.

#### Стратегия 4: Плановая ротация ключей (рекомендуется)

Проектировать систему так, чтобы ротация ключей была **штатной операцией**:

1. Каждая версия enclave получает свой keypair
2. SignerListSet временно расширяется (3 старых + 1 новый, quorum
   **поднимается до 3**). Это предотвращает ситуацию, когда 1
   скомпрометированный старый ключ + 1 новый = quorum. После удаления
   старого signer'а quorum возвращается к 2.
3. Rolling upgrade: одна нода за раз
4. Автоматизировано CLI:
   ```
   orchestrator enclave-upgrade \
     --node node-1 \
     --new-enclave /path/to/enclave.signed.so \
     --escrow-address rLTFG...
   ```
   Команда:
   a. Запуск нового enclave на staging-порту
   b. Генерация нового keypair → новый XRPL-адрес
   c. Новый enclave генерирует DCAP attestation quote, привязывающий
      MRENCLAVE к новому публичному ключу
   d. Каждый существующий enclave независимо проверяет DCAP quote
      перед тем как согласиться подписать SignerListSet
   e. SignerListSet (добавить нового, **поднять quorum до 3**)
   f. Ожидание XRPL валидации (ограничено `LastLedgerSequence`)
   g. SignerListSet (удалить старого, **вернуть quorum на 2**)
   h. Остановка старого enclave, безопасное удаление sealed данных
   i. Промоция нового enclave на рабочий порт

**Единственная стратегия, сохраняющая модель безопасности SGX.**

#### Анализ поверхности атаки

Ротация ключей через SignerListSet — это по сути смена владельца.
Исторически это главный вектор атак на любой мультисиг. Каждый вектор
ниже имеет конкретный механизм защиты; вектор без защиты неприемлем.

**Вектор 1: Кража quorum в окне 4 signer'ов**

*Атака:* во время ротации в signer list 4 записи. Если quorum остаётся
2, атакующий с 1 скомпрометированным старым ключом + 1 новым получает
quorum и может вывести средства из эскроу.

*Защита:* quorum **поднимается до 3** на время окна ротации (шаг e).
При 4 signer'ах и quorum=3 атакующему нужно 3 ключа — строго сложнее
чем обычные 2-of-3. Quorum возвращается к 2 только после удаления
старого signer'а и возврата списка к 3 записям.

**Вектор 2: Подмена нового ключа**

*Атака:* оркестратор (или MITM) подменяет реальный публичный ключ
нового enclave на адрес атакующего в SignerListSet транзакции.

*Защита:* новый enclave генерирует keypair внутри себя и создаёт
**DCAP attestation quote**, привязывающий хеш MRENCLAVE к новому
публичному ключу. Каждый из 3 существующих enclave'ов независимо
верифицирует этот quote перед подписью SignerListSet. Оркестратор
предлагает транзакцию, но не может подделать подписи enclave'ов — ему
нужно согласие 2-of-3 существующих enclave'ов, и каждый проверяет DCAP.
Подменённый ключ не пройдёт верификацию quote ни на одном честном enclave.

**Вектор 3: Ошибка удаления старого signer'а (застрявший ключ)**

*Атака:* шаг g (удалить старого) не выполняется — таймаут сети, скачок
комиссий, падение оркестратора. Старый ключ остаётся в signer list
бессрочно, расширяя поверхность атаки до 4 ключей вместо 3.

*Защита:*
1. `LastLedgerSequence` на add-signer tx: если не подтверждена за N
   леджеров (~20 секунд), вся миграция отменяется, ключ не добавлен.
2. После успешного добавления: remove-signer tx повторяется с
   экспоненциальным backoff до подтверждения. Оркестратор отказывается
   считать миграцию завершённой, пока on-chain signer list не вернётся
   ровно к 3 записям.
3. **Жёсткий таймаут:** если remove-signer не подтверждён за 10 минут,
   система откатывается — отправляет SignerListSet, удаляющий НОВЫЙ ключ,
   возвращаясь к исходному 3-of-3. Лучше прервать апгрейд, чем оставить
   окно из 4 ключей открытым.
4. Health check мониторит количество signer'ов: любое число ≠ 3 —
   алерт.

**Вектор 4: Переупорядочивание SignerListSet транзакций**

*Атака:* две транзакции (добавить нового, удалить старого) приходят не
в том порядке, или вторая валидируется пока первая ещё в pending.

*Защита:* XRPL обеспечивает строгий порядок по `Sequence` для каждого
аккаунта. Remove-signer tx имеет Sequence = add-signer Sequence + 1,
поэтому она **не может** валидироваться раньше add-signer. Если первая
tx отброшена, вторая автоматически невалидна. Это гарантия протокола
XRPL, а не логика приложения.

**Вектор 5: Компрометация оркестратора (единая точка доверия)**

*Атака:* оркестратор конструирует SignerListSet транзакции и общается со
всеми enclave'ами. Скомпрометированный оркестратор может предложить
добавить ключ атакующего.

*Защита:* оркестратор конструирует, но не подписывает — подписывают
только enclave'ы. Добавление нового signer'а требует подписи 2-of-3
существующих enclave'ов. Каждый enclave верифицирует DCAP quote нового
ключа перед подписью (защита от Вектора 2). Скомпрометированный
оркестратор может предложить вредоносный ключ, но enclave'ы его
отклонят — DCAP quote не пройдёт верификацию. Оркестратор — координатор,
а не корень доверия.

**Вектор 6: Откат к старому enclave после ротации**

*Атака:* после успешной ротации атакующий откатывает систему к старому
enclave (у которого ещё есть старые sealed ключи). Если старый ключ не
был удалён из SignerListSet, атакующий контролирует валидного signer'а.

*Защита:*
1. Шаг h явно удаляет старые sealed данные (`shred` + `rm` директории
   аккаунтов и sealed blob'ов старого enclave).
2. Старый `enclave.signed.so` удаляется с диска.
3. Даже если старый бинарник восстановлен, sealed данные удалены — у
   старого MRENCLAVE нет ключей для расшифровки.
4. On-chain: старый адрес удалён из SignerListSet, поэтому даже если
   старый ключ каким-то образом существует, он не может участвовать
   в подписании.

**Вектор 7: Параллельная ротация на нескольких нодах**

*Атака:* две ноды ротируются одновременно, создавая конфликтующие
SignerListSet транзакции (обе пытаются расширить 3→4, результат — 5
signer'ов или неконсистентное состояние).

*Защита:* строго последовательная ротация, обеспечиваемая CLI
оркестратора. Инструмент принимает `--node` и ротирует ровно одну
ноду за вызов. Второй вызов проверяет on-chain количество signer'ов —
если не ровно 3 (предыдущая ротация в процессе или не завершена),
отказывается стартовать.

**Вектор 8: DoS во время окна ротации**

*Атака:* атакующий не даёт ротации завершиться (сетевая партиция,
перегрузка XRPL, падение enclave), удерживая систему в состоянии
повышенного quorum с 4 signer'ами. Хотя это не прямая кража, это
деградирует систему — quorum=3 означает, что все 3 старых + новый
должны кооперироваться, что усложняет штатные операции.

*Защита:* жёсткий таймаут из Вектора 3 применяется: если полная ротация
(добавить + удалить) не завершена за 10 минут, система автоматически
откатывается — удаляет новый ключ и восстанавливает quorum=2. Ротацию
можно повторить позже в стабильных условиях. Во время окна повышенного
quorum штатные операции (выводы) продолжают работать — им просто нужно
3 подписи вместо 2, что строже, но функционально.

### Стратегия 1 vs Стратегия 4: детальное сравнение

Стратегии 1 и 4 разделяют один и тот же базовый механизм — сгенерировать
ключ в новом enclave, добавить в SignerListSet, удалить старый. Разница
не в том *что* они делают, а в том *сколько защитных механизмов окружают
операцию*.

| Свойство | Стратегия 1 (dual-server) | Стратегия 4 (плановая ротация) |
|----------|---------------------------|--------------------------------|
| **Исполнение** | Ручное — оператор выполняет шаги | Автоматическое — одна CLI команда |
| **Quorum при ротации** | Без изменений (остаётся 2-of-4) | Поднимается до 3-of-4 на время окна |
| **Верификация нового ключа** | Нет — оператор визуально подтверждает адрес | DCAP quote: каждый существующий enclave криптографически верифицирует что новый ключ принадлежит реальному enclave с известным MRENCLAVE |
| **Таймаут / откат** | Нет — если шаг 4 не выполнен, оператор должен заметить и вмешаться | Жёсткий 10-минутный таймаут; авто-откат удаляет новый ключ, восстанавливает quorum=2 |
| **Очистка старого ключа** | Ответственность оператора — может забыть | Автоматический shred+rm sealed данных и старого бинарника |
| **Защита от параллельной ротации** | Нет — оператор должен помнить "по одной" | CLI проверяет on-chain signer count, отказывается стартовать если ≠ 3 |
| **Порядок транзакций** | Оператор отправляет вручную — может ошибиться | Автоматическая нумерация Sequence + ограничение LastLedgerSequence |
| **Доверие к оркестратору** | Оператор И ЕСТЬ корень доверия — он выбирает ключ и отправляет tx | Оркестратор — только координатор; enclave'ы верифицируют DCAP перед подписью |
| **Мониторинг signer count** | Ручной — оператор проверяет XRPL explorer | Health check алерт при signer count ≠ 3 |
| **Время на 3 ноды** | ~15 мин (зависит от оператора) | ~15 мин (автоматически, та же задержка XRPL) |
| **Стоимость реализации** | Низкая — только runbook | Высокая — DCAP верификация в enclave, CLI тулинг, логика отката |
| **Работает без DCAP?** | Да — нет шага аттестации | Деградировано — без DCAP защита от Вектора 2 (подмена ключа) ослаблена |

#### Когда Стратегия 1 — правильный выбор

Стратегия 1 — это не "ухудшенная версия" Стратегии 4. Это правильный
выбор когда:

1. **DCAP недоступен.** На Hetzner (SGX1, нет DCAP) защита Вектора 2
   (верификация нового ключа через DCAP) не работает. Стратегия 1 с
   доверенным оператором, который вручную верифицирует ключ нового
   enclave — единственный вариант. Оператор компенсирует отсутствие
   криптографической верификации физическим доступом + визуальным
   подтверждением.

2. **Автоматика Стратегии 4 сломалась.** Если CLI крашнулся, логика
   отката не сработала, или система в неожиданном состоянии — Стратегия 1
   является ручной процедурой восстановления. Любая автоматическая
   система нуждается в ручном fallback.

3. **Первая ротация на testnet.** Прежде чем доверять автоматике,
   выполните процедуру вручную для понимания операций. Невозможно
   дебажить автоматическую ротацию, если ни разу не делал её руками.

#### Когда Стратегия 4 обязательна

Стратегия 4 обязательна когда:

1. **На кону реальные средства.** Ручные процедуры имеют процент
   человеческих ошибок. Подъём quorum (2→3 в окне), жёсткий таймаут
   и автоматический откат существуют именно потому, что человек-оператор
   может забыть шаг, отвлечься или стать жертвой социальной инженерии.

2. **Несколько операторов.** При 3 независимых операторах Стратегия 1
   требует координации («я ротирую свою ноду, не ротируй свою»).
   Стратегия 4 обеспечивает это на уровне протокола — CLI проверяет
   on-chain состояние.

3. **Важен аудит-трейл.** Стратегия 4 создаёт машинно-верифицируемый
   лог: DCAP quote, SignerListSet транзакции с Sequence, метки времени,
   события отката. Стратегия 1 создаёт «оператор говорит что всё
   сделал правильно».

#### Связь между стратегиями

Стратегия 4 = Стратегия 1 + подъём quorum + DCAP верификация +
автоматический таймаут/откат + проверка signer count + очистка sealed
данных.

Стратегия 1 — это *вырожденный случай* Стратегии 4, где каждый
защитный механизм заменён на «оператор сделает правильно». Это
приемлемо для testnet и аварийных ситуаций. Это неприемлемо для mainnet
с реальными средствами и независимыми операторами, которые не полностью
доверяют друг другу — а это и есть весь смысл мультисига.

### Рекомендация

| Фаза | Стратегия | Почему |
|------|-----------|--------|
| **Сейчас** (testnet) | Стратегия 1 (dual-server) | Ручная но безопасная; тренируемся на testnet. Нарабатываем мышечную память. |
| **До mainnet** | Построить тулинг Стратегии 4 | Должен быть готов до mainnet с реальными деньгами |
| **Mainnet** | Стратегия 4 (rolling key rotation) | Единственный вариант, сохраняющий trust model SGX в условиях adversarial среды |
| **Аварийно** | Стратегия 1 (dual-server) | Ручное восстановление когда автоматика Стратегии 4 сломалась |
| **Hetzner (нет DCAP)** | Стратегия 1 с доверенным оператором | DCAP недоступен — оператор компенсирует физической верификацией |

### MVP-этап: практическая схема (текущая стадия)

**Важно для коллег:** Стратегия 4 — это долгосрочная цель, а не то, что
нужно внедрять прямо сейчас. На этапе MVP — с ~108 XRP в эскроу,
трафиком около нуля, одним доверенным оператором и LLM-ассистентом,
выполняющим команды под контролем оператора — Стратегия 1 является
правильным и достаточным подходом.

#### Двойная схема на Hetzner EX44

Проблема деплоя (Варианты A–E выше) решается для MVP-этапа запуском
двух полностью изолированных инстансов на существующем Hetzner:

```
Hetzner EX44 (94.130.18.162)
├── MAINNET (продакшен)
│   ├── Enclave:      порт 9088, SGX HW mode
│   ├── Оркестратор:  порт 3000, подключен к XRPL mainnet
│   ├── Sealed data:  /tmp/perp-9088/
│   └── БД:           perp_dex (PostgreSQL)
│
└── DEV/STAGING (разработка)
    ├── Enclave:      порт 9089, SGX HW mode
    ├── Оркестратор:  порт 3001, подключен к XRPL testnet
    ├── Sealed data:  /tmp/perp-9089/
    └── БД:           perp_dex_dev (PostgreSQL)
```

Это **не** Вариант C из раздела деплоя (где один enclave на двоих). Это
два полностью отдельных процесса enclave с отдельными sealed данными,
отдельными базами и отдельными XRPL сетями. Единственный общий ресурс —
физическая машина.

**Риск:** OOM или CPU-спайк от dev-инстанса может уронить mainnet.
Приемлемо при текущем трафике (~0 TPS, ~108 XRP). Неприемлемо когда
средства вырастут — тогда переезд на второй Hetzner (Вариант A) или
EX130 (Вариант D).

#### Процедура обновления enclave на MVP-этапе

Это Стратегия 1, выполняемая LLM-ассистентом под контролем оператора:

1. **Сборка** нового `enclave.signed.so` на Hetzner из git
2. **Деплой на dev** (порт 9089) — новый enclave получает новый MRENCLAVE
3. **Генерация нового ключа** в dev enclave → новый XRPL testnet адрес
4. **Тест ротации на testnet** — SignerListSet добавить нового signer'а,
   затем удалить старого. Проверить в XRPL testnet explorer.
5. **Повтор на mainnet** (порт 9088) только после успеха на testnet:
   - Деплой нового enclave на временный порт (9090)
   - Генерация нового ключа → новый mainnet адрес
   - SignerListSet: добавить нового signer'а в mainnet эскроу
   - Проверка on-chain
   - SignerListSet: удалить старого signer'а
   - Остановка старого enclave (9088), промоция нового на порт 9088
   - Удаление старых sealed данных

**Оператор видит каждую команду перед выполнением и может прервать на
любом шаге.** Это trust model Стратегии 1: оператор — корень доверия,
а не криптографическая верификация.

#### Когда выходить за рамки этой схемы

| Сигнал | Действие |
|--------|----------|
| Баланс эскроу превышает 10 000 XRP | Перенести dev на отдельный сервер (Вариант A или D) |
| Появляется второй независимый оператор | Начать разработку тулинга Стратегии 4 |
| Грантовый аудит требует audit trail | Начать разработку тулинга Стратегии 4 |
| Azure VM больше не нужны PM | Деаллоцировать Azure, сэкономить €300/мес |

### Что это значит для deployment dilemma

Проблема enclave upgrade **усиливает** проблему Hetzner-vs-Azure:

1. **Hetzner работает в SGX HW mode** — MRENCLAVE binding реальный,
   sealing работает, ключи привязаны к железу. Обновление enclave на
   Hetzner потеряет доступ к sealed ключам точно так же как на Azure.
2. **Hetzner НЕ поддерживает DCAP аттестацию** — нет PCK provisioning,
   нет PCCS. Процедура обновления enclave (ротация ключей) тестируется
   на Hetzner, но attestation-зависимые флоу — только на Azure DCsv3.
3. **Обновление enclave МОЖНО тестировать на Hetzner** — ротация ключей
   (dual-server, SignerListSet swap) работает идентично. Но тестировать
   на mainnet с живыми деньгами рискованно — нужен staging (Hetzner №2
   или одна Azure VM).

**Аргумент за сохранение хотя бы одной Azure VM** — конкретно для
тестирования DCAP аттестации, а не для обновления enclave, которое
можно репетировать на любом SGX-железе включая Hetzner.
