## Процедура обновления mainnet на Hetzner

Чеклист безопасного обновления:

1. **Сборка** на Hetzner из git (staging-ветка протестирована)
2. **Снимок** текущего состояния:
   - `cp -r /tmp/perp-9088/ /tmp/perp-9088.backup-$(date +%s)/`
   - `pg_dump perp_dex > /tmp/perp_dex_backup.sql`
3. **Остановка** mainnet: `kill <PID>` (nginx отдаёт 502)
4. **Замена** бинарника: `cp target/release/perp-dex-orchestrator /tmp/perp-9088/`
5. **Запуск** с теми же аргументами
6. **Health check**: `curl http://localhost:3000/v1/health`
7. **Проверка** XRPL mainnet: баланс эскроу не изменился
8. **Откат** если health check не прошёл:
   - `kill <PID>`
   - Восстановить бинарник из backup
   - Перезапуск

**Даунтайм:** ~30 секунд. Приемлемо при текущем трафике (около нуля).

## Сложный вопрос: обновление Enclave

Всё выше обсуждает деплой **оркестратора** (Rust). Enclave (C++ SGX) —
фундаментально сложнее из-за **привязки MRENCLAVE**.

### Почему enclave ≠ оркестратор

| Свойство | Оркестратор (Rust) | Enclave (C++ SGX) |
|----------|--------------------|--------------------|
| Состояние | PostgreSQL (внешнее) | Sealed к CPU + MRENCLAVE |
| Замена бинарника | Stop → replace → start | Новый MRENCLAVE → **все sealed ключи потеряны** |
| Откат | Скопировать старый бинарник | Sealed данные старого MRENCLAVE работают |
| Ключи | Живут в enclave, оркестратор не имеет | Ключи **и есть** состояние enclave |
| Плохой деплой | 502 на 30 сек | **Безвозвратная потеря ключей** |

### Проблема MRENCLAVE

SGX `sgx_seal_data()` привязывает данные к:
1. **MRENCLAVE** — хеш бинарника (меняется при ЛЮБОМ изменении кода)
2. **Идентификатор CPU** — per-platform ключ

Это означает:
- Новый `enclave.signed.so` (даже исправление опечатки) = **другой MRENCLAVE**
- Новый enclave **не может расшифровать** данные старого
- Приватные ключи **безвозвратно потеряны** для нового enclave
- Это by design в SGX, не баг

### Что теряется при наивном обновлении

1. Все приватные ключи enclave → XRPL-адреса из SignerListSet
2. Мультисig эскроу перестаёт работать
3. Средства пользователей заблокированы
4. FROST shares (если используются) — не переносимы между машинами

### Когда ПРИДЁТСЯ обновлять enclave

Известные триггеры (не гипотетические):

1. **FROST cross-machine share transport** — сломан (Bug 1 в
   SHARED-ENCLAVE-BUGS.md). Фикс = C++ → новый MRENCLAVE
2. **BTC signing path** — если PM понадобится
3. **`MAX_FROST_GROUPS = 4`** — слишком мало для продакшена
4. **DKG "share already received" без reset** — блокирует retry
5. **Патчи безопасности** — любой CVE в коде enclave
6. **Новые фичи** — spending limits, margin engine внутри enclave

### Стратегии обновления

#### Стратегия 1: Dual-Server миграция

Старый enclave (порт 9088) и новый (порт 9089) работают параллельно.

**Процедура:**
1. Деплой нового enclave на отдельный порт
2. Генерация нового ключа → новый XRPL-адрес
3. Старый enclave подписывает SignerListSet: добавить нового signer'а
4. Старый enclave подписывает SignerListSet: удалить старого
5. Старый enclave деактивируется

**Критично:** старый enclave должен оставаться рабочим на протяжении
всей миграции.

#### Стратегия 2: MRSIGNER-based Sealing (слабее, но проще)

Поменять политику sealed с MRENCLAVE на MRSIGNER — любой enclave,
подписанный тем же ключом вендора, может расшифровать данные.

| + | – |
|---|---|
| Zero-downtime апгрейд | Слабая безопасность: скомпрометированный пайплайн = извлечение ключей |
| Без миграции ключей | Аудиторы это отметят |
| Просто: заменил бинарник, рестарт | Теряется гарантия привязки к коду |

**Вердикт:** приемлемо для testnet, НЕ для mainnet.

#### Стратегия 3: Экспорт через recovery

Использовать `account.recovery` файлы для экспорта/импорта ключей.

**Проблема:** оператор кратковременно получает доступ к raw private key
вне enclave — нарушение trust model.

**Вердикт:** только если recovery key сам в HSM/enclave.

#### Стратегия 4: Плановая ротация ключей (рекомендуется)

Проектировать систему так, чтобы ротация ключей была **штатной операцией**:

1. Каждая версия enclave получает свой keypair
2. SignerListSet временно расширяется (3 старых + 1 новый, quorum = 2)
3. Rolling upgrade: одна нода за раз
4. Автоматизировано CLI:
   ```
   orchestrator enclave-upgrade \
     --node node-1 \
     --new-enclave /path/to/enclave.signed.so \
     --escrow-address rLTFG...
   ```
   Команда: запуск нового enclave → генерация ключа → SignerListSet
   (добавить) → ожидание → SignerListSet (удалить старого) → остановка
   старого → промоция нового.

**Единственная стратегия, сохраняющая модель безопасности SGX.**

### Рекомендация

| Фаза | Стратегия | Почему |
|------|-----------|--------|
| **Сейчас** (testnet) | Стратегия 2 (MRSIGNER) | Быстрая итерация, нет реальных средств |
| **До mainnet** | Построить тулинг Стратегии 4 | Должен быть готов до mainnet с реальными деньгами |
| **Mainnet** | Стратегия 4 (rolling key rotation) | Единственный вариант с сохранением trust model |
| **Аварийно** | Стратегия 1 (dual-server) | Ручная, но безопасная, когда автоматика сломалась |

### Что это значит для deployment dilemma

Проблема enclave upgrade **усиливает** проблему Hetzner-vs-Azure:

1. **Hetzner работает в SGX HW mode** — MRENCLAVE binding реальный,
   sealing работает, ключи привязаны к железу. Обновление enclave на
   Hetzner потеряет доступ к sealed ключам точно так же как на Azure.
2. **Hetzner НЕ поддерживает DCAP аттестацию** — нет PCK provisioning,
   нет PCCS. Процедура обновления enclave (ротация ключей) тестируется
   на Hetzner, но attestation-зависимые флоу — только на Azure DCsv3.
3. **Обновление enclave МОЖНО тестировать на Hetzner** — ротация ключей
   (dual-server, SignerListSet swap) работает идентично. Но тестировать
   на mainnet с живыми деньгами рискованно — нужен staging (Hetzner №2
   или одна Azure VM).

**Аргумент за сохранение хотя бы одной Azure VM** — конкретно для
тестирования DCAP аттестации, а не для обновления enclave, которое
можно репетировать на любом SGX-железе включая Hetzner.
